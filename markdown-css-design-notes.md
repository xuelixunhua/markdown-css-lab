# Markdown CSS 主题整理笔记

这次整理的目标不是再收集一堆零散样式，而是形成一套可维护的 Markdown 渲染系统：同一套审美原则，根据场景分成 PDF 手册、公众号长文、技术文档三种输出。

## 1. 新整理的三套 CSS

- `themes/founder-editorial.css`：PDF / 手册 / 研究笔记导出。核心是 A4 版心、衬线标题、无衬线正文、暖陶土强调色、浅色 callout。
- `themes/wechat-editorial.css`：公众号排版。核心是 `#nice { padding: 0 }`、移动端 16px 正文、陶土强调色、克制标题和引用。
- `themes/technical-doc.css`：技术文档。核心是 GitHub 式信息密度、中文友好的字体栈、代码块、表格、提示块和打印样式。

这三套主题都没有覆盖原文件。原文件仍然可以作为素材库：`latex.css` 适合继续吸收打印和编号能力，`github.css` 适合吸收技术元素，`公众号.css` 适合吸收微信端的容器经验。

## 2. 市场情况简述

Markdown CSS 生态大致分四类：

1. GitHub 风格：`github-markdown-css` 是代表，定位是用尽量少的 CSS 复刻 GitHub Markdown 样式，并提供多套明暗主题。适合技术文档、README、API 文档。
   Source: https://github.com/sindresorhus/github-markdown-css

2. Typora 主题：Typora 官方支持 CSS 主题，社区主题很多，GitHub topic 下能看到 LaTeX、Notion、GitBook、Misty、Mdmdt、Claude-like 等方向。适合本地写作和 PDF 导出。
   Sources: https://support.typora.io/About-Themes/ , https://github.com/topics/typora-theme , https://theme.typora.io/theme/Claude-Theme/

3. 通用 Markdown CSS：如 Markdown CSS collection，提供 Splendor、Air、Modest 等不同视觉方向。适合作为灵感库，但中文长文和 PDF 细节通常要二次改。
   Source: https://markdowncss.github.io/

4. 公众号 Markdown 编辑器：mdnice、doocs/md、WeMD 等都围绕 Markdown -> 微信公众号 HTML/CSS。它们的价值在于兼容微信公众号编辑器、内联样式、自定义主题、深色预览或本地编辑。
   Sources: https://github.com/mdnice/markdown-nice , https://github.com/doocs/md , https://wemd.app/

还有一类是 Pandoc / print CSS。它们更重视响应式、旁注、目录、打印样式、超宽表格和代码块处理。对我们做 PDF 手册很有参考价值。
Source: https://jez.io/pandoc-markdown-css-theme/

结论：市场上工具很多，但大多只解决一个端点。我们的机会是把“中文 Markdown 长文”统一成一个审美系统，再按输出端分皮肤。

## 3. 版式基础知识

### (a) 什么是强版心

版心就是页面里正文真正占据的矩形区域。强版心不是说边框很明显，而是正文、标题、表格、引用、页码都稳定地服从同一个 invisible grid。

好看的 PDF 往往不是每个元素都好看，而是所有元素都知道自己该站在哪里。你给的 PDF 正文页内容大致固定在左 56.7pt 到右 538.6pt，换算约为左右各 20mm，内容宽度约 170mm。这就是强版心。

### (b) 20mm 页边距是什么意思，多少最完美

A4 宽 210mm。左右各 20mm 后，正文宽度约 170mm。这个宽度适合手册型 PDF：既不会像论文那样窄，也不会像网页那样散。

你的现有 CSS 不完全一样：

- `latex.css`：屏幕下 `padding: 2.54cm 1.18cm`，打印下左右 `3.18cm`，更像 Word/论文，边距比 PDF 更宽。
- `github.css`：`max-width: 860px; padding: 30px`，是网页逻辑，不是毫米版心。
- `公众号.css`：`#nice { padding: 0 }` 和 `p { margin: 10px 0 }`，是移动端逻辑，核心是减少左右空白。

没有最完美的固定值，只有用途：

- PDF 手册：18-24mm。
- 技术白皮书：16-22mm。
- 学术论文：25-32mm。
- 公众号：容器 padding 尽量 0-8px，正文靠平台屏幕宽度自然收缩。

### (c) 衬线和无衬线

衬线字体是笔画末端带小脚、小装饰的字体，如宋体、Noto Serif CJK、Times、Georgia。它更有书卷感、正式感、标题感。

无衬线字体是笔画末端干净的字体，如黑体、PingFang、Noto Sans、Microsoft YaHei、Arial。它更现代、清晰、适合界面和正文。

目前你的 CSS 有一些区分，但不稳定：

- `latex.css` 最明确：正文用 LM Roman / Source Han Serif，标题部分用宋体/黑体组合。
- `github.css` 基本是无衬线技术文档。
- `公众号.css` 用 Optima/PingFang，更偏轻文艺但没有明确“标题衬线、正文无衬线”的系统。

这可以作为排版原则：标题用衬线建立气质，正文用无衬线保证长读舒适，代码用等宽字体保持结构。

### (d) 颜色怎么用

建议建立语义色，而不是随手写颜色：

- `ink`：正文黑，例如 `#1a1a1a`。
- `muted`：英文副标题、说明文字，例如 `#777` 或 `#888`。
- `rule`：分割线和表格线，例如 `#e7dfd8`。
- `tint`：浅底提示块，例如 `#faf6f2`。
- `accent`：唯一主强调色，例如 PDF 的陶土色 `#d87556`。
- `accent-dark`：链接、重点文字，例如 `#6d3528`。

公众号里不要大面积使用高饱和红。原来的 `#ff3502` 很醒目，但容易显得促销和刺眼。更建议用 `#d87556` 做视觉记忆，用深陶土色做文字强调。

### (e) 正文密度：手册和公众号的区别

手册型 PDF：

- 正文：10.5-11pt，行高 1.65-1.75。
- H1：32-38pt，用于章标题或封面。
- H2：20-24pt。
- H3：14-16pt。
- 段落间距：0.8-1rem，克制。
- 观感：连续、稳定、像书。

公众号推文：

- 正文：16px，行高 1.65-1.8。
- H1：24-26px。
- H2：20-22px，常居中或带短装饰线。
- H3：17-18px，可用左边线增强扫描。
- 段落间距：10-14px。
- 观感：手机上更松，段落更短，强调更明显。

技术文档：

- 正文：16px。
- H1/H2 带底线。
- 代码、表格、列表要优先清晰。
- 观感：密度高、可扫描、少情绪。

### (f) 五级标题怎么处理

建议 Markdown 标题不要都做成“大标题”。五级标题应该分工：

- H1：文档标题或大章节。PDF 可分页；公众号一篇文章只用一次。
- H2：主要章节。适合居中或大号衬线。
- H3：小节标题。适合左对齐、加粗、可带左边线。
- H4：段内主题。像“挑战：”“解决：”这种局部结构。
- H5：标签型标题。小字号、强调色、可大写或半标签化。
- H6：注释型标题。灰色、很小，只用于补充说明。

真正的层级感来自字号、字体、颜色、间距四个变量一起变化，而不是每一级都加装饰。

## 4. 你的公众号边距代码哪里起作用

起决定作用的是这几处：

```css
#nice {
  padding: 0px;
}
```

它去掉了 `#nice` 容器的内边距。公众号 Markdown 工具通常会把文章包在 `#nice` 里，容器 padding 一大，整篇文章左右都会缩进去。

```css
#nice p {
  margin: 10px 0px;
}
```

这里的 `0px` 是左右 margin。也就是说段落本身不再额外制造左右空白。上下保留 10px，让段落仍然有呼吸。

```css
#nice blockquote {
  margin: 20px 0px;
}
```

引用块也取消左右 margin，否则引用会比正文再窄一圈。

```css
#nice blockquote p {
  margin: 10px;
}
```

这行只影响引用块内部文字，让引用内部保留 10px 缓冲。外面不缩，里面有呼吸。

```css
#nice h2 {
  margin: 50px 10px 25px 10px;
}
```

这会让二级标题左右各保留 10px，所以它不是“全局缩边距”的核心，只是让标题别贴得太死。

另外：

```css
text-align: justify;
word-break: break-all;
```

这两行不是边距控制。`text-align: justify` 会让段落两端更齐，视觉上显得更满；`word-break: break-all` 会避免英文长词撑破容器，但会破坏英文单词。新版建议改成 `overflow-wrap: break-word` 或 `overflow-wrap: anywhere`，少用 `break-all`。

这部分已经吸收到 `themes/wechat-editorial.css`：保留 `#nice { padding: 0 !important }`、段落左右 margin 为 0、引用外边距左右为 0，但移除了 `word-break: break-all`。

## 5. 三套 CSS 的逻辑对比

### 总体定位

`founder-editorial.css` 的关键词是“手册感”。它服务 PDF、研究笔记、小册子，追求稳定版心、强留白、衬线标题和少量品牌色。

`wechat-editorial.css` 的关键词是“手机长文”。它服务公众号，目标是在窄屏上读起来不憋、不散，重点能跳出来，但不能像广告海报。

`technical-doc.css` 的关键词是“扫描效率”。它服务 README、API 文档、技术规范，优先让标题、代码、表格、提示块快速被识别。

### 标题系统

PDF 手册主题把 H1/H2/H3 交给衬线字体，营造“书”的气质；H4 以下回到无衬线，承担局部结构。它的标题不是为了抢眼，而是为了建立章节节奏。

公众号主题让 H2 居中并加短横线，H3 用左边线。这样手机上滑动时，读者能马上看到结构停顿。H1 控制为一篇文章标题，不鼓励频繁出现。

技术文档主题沿用 GitHub 式 H1/H2 底线。原因是技术文档需要“可定位”，底线比装饰更实用，读者可以快速分辨章节边界。

### 正文密度

PDF 手册主题正文约 11pt、行高 1.72，段落间距克制。它适合连续阅读，像一本排过版的说明书。

公众号主题正文 16px、行高 1.75，段落上下 12px。手机上需要更松一点，因为屏幕窄、阅读距离近，段落要更容易被眼睛切开。

技术文档主题正文 16px、行高 1.65，段落和列表更紧。它默认读者不是线性阅读，而是在查找、比较、复制代码。

### 目录 TOC

PDF 手册主题的目录现在改成紧凑清单：小字号、低行高、无横线。目录的任务是给全局地图，不应该占太多页面。

公众号主题的目录保留浅底和左侧强调线，但缩小 padding、字号和行距。因为公众号目录通常只是文章开头的导航，不宜做成大卡片。

技术文档主题的目录仍然保留轻边框和浅底，因为技术文档里 TOC 是功能性模块，但行距也收紧，避免压缩正文空间。

### 颜色系统

PDF 手册主题只使用一个暖陶土主色 `#d87556`，正文基本是黑灰。大面积颜色只给封面和章节页，正文里只做细线、浅底、引用。

公众号主题同样继承陶土色，但加粗文字单独使用更鲜明的 `#d94721`。手机屏幕上正文黑和深陶土差异不够，所以重点需要略微更亮、更热。

技术文档主题不用陶土作为核心，而是使用蓝、绿、橙、红这些功能色。原因是技术文档里的颜色更像状态语言：note、tip、warning、danger。

### 引用和提示块

PDF 手册主题的引用块和 callout 都很克制：浅暖底加左边线。它像书里的旁注，而不是网页组件。

公众号主题的引用块稍微更醒目，但仍然不加阴影。公众号里阴影和圆角过多容易显得模板化。

技术文档主题把提示块分成 note、tip、warning、danger。这里的目标不是美学情绪，而是风险和信息类型的识别。

### 表格和代码

PDF 手册主题的表格更接近三线表，顶部强调线较强，适合研究笔记和结论型表格。

公众号主题的表格尽量简单，字号略小，避免在手机上撑爆。

技术文档主题保留横向滚动、代码块浅灰底和等宽字体，优先保证复制、阅读、对齐。

### 设计取舍

三套主题共享同一套底层原则：稳定版心、明确层级、少量语义色、减少无意义装饰。但三者的“阅读姿势”不同，所以不能做成同一张皮。

PDF 手册是坐下来读，公众号是滑动中读，技术文档是带着问题查。标题、目录、正文密度、颜色强调都围绕这个差异展开。
